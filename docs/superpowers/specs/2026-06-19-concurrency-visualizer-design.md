# trace-go — Go 동시성 시각화 도구 설계

- **날짜**: 2026-06-19
- **상태**: 설계 승인 대기
- **목표**: Go 동시성(goroutine/channel)을 *눈으로 이해하는* 데스크톱 시각화 도구

---

## 1. 개요 & 범위

### 정체성
`trace-go`는 Go 동시성을 직관적으로 이해하도록 돕는 데스크톱 시각화 도구다.
`go tool trace`가 사용하는 것과 동일한 공식 실행 추적 데이터(`runtime/trace`)를,
학습자와 실무자 모두에게 직관적인 **하이브리드 뷰**(타임라인 스크러버 + 시점별 라이브 goroutine/channel 그래프)로 재해석한다.

참고로 영감을 받은 [divan/gotrace](https://github.com/divan/gotrace)는 **Go 런타임을 직접 패치**해 이벤트를 수집했고,
이 때문에 Go 버전 업마다 깨져 유지보수가 불가능해졌다. `trace-go`는 이 실수를 피하기 위해
런타임 패치 없이 **공식적으로 안정적인 `runtime/trace` 데이터**만 사용한다.

### 1차 타겟 사용자
학습자(동시성 개념을 처음 익히는 사람)와 실무자(자기 프로그램의 동시성 동작을 이해·디버깅하려는 사람) **둘 다**.
단, v1은 아래 범위로 단계적으로 충족한다.

### v1 범위 (In scope)
- 사용자가 `runtime/trace` 또는 `go test -trace`로 생성한 **기존 trace 파일을 "열기"로 로드**하는 순수 뷰어
- `golang.org/x/exp/trace` 기반 파싱 → 내부 공통 이벤트 모델
- **하이브리드 시각화**: 상단 타임라인 스크러버 + 하단 시점별 라이브 그래프, 재생/일시정지/스크럽
- goroutine 생명주기·블록(이유)·**언블록 인과 엣지**(채널 send↔recv 추정) 표시
- **중간 규모(수백 goroutine)** 를 Canvas 2D로 부드럽게 렌더
- Wails 데스크톱 앱(단일 바이너리)
- **Go 1.22+ 새 trace 포맷 전용** (인과 정보가 핵심이며 구 포맷은 이를 살리지 못함)

### 비범위 (Non-goals, 후순위)
- 앱이 프로그램을 직접 빌드·실행해주는 turnkey 모드 → v2
- 대규모(수천+ goroutine) WebGL 렌더링 → 렌더 계층만 추상화해두고 후속
- 라이브러리 계측 방식·정적 분석·실시간 스트리밍
- 채널 고유 식별 / 전달 값 복원 (trace 데이터로 불가능 — 인과 추정만 제공)
- Go 1.22 미만 구 trace 포맷

## 2. 아키텍처

Wails 구조(Go 백엔드 ↔ JS 프론트엔드, bridge 통신)에 맞춰 책임을 분리한다.

```
┌─────────────────────────── Wails App ───────────────────────────┐
│                                                                  │
│  Go 백엔드 (parsing & model)        JS 프론트 (Svelte + Canvas)  │
│  ───────────────────────────        ──────────────────────────  │
│  • OpenTrace(path) 바인딩            • 파일 열기 UI / 컨트롤바    │
│  • trace 파서                        • TimelineCanvas (상단)      │
│    (x/exp/trace 래핑)                • GraphCanvas (하단, 라이브) │
│  • 이벤트 정규화 → core model        • 재생/스크럽 상태 store     │
│  • 인과 엣지 재구성                  • 시점 t → 표시 상태 계산    │
│  • 시각화용 요약 데이터 생성  ──JSON──▶  (프레임 렌더)           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 책임 경계 (핵심 원칙: 무거운 계산은 Go, 그리기는 JS)
- **Go 백엔드**는 trace 파싱·이벤트 정규화·인과 엣지 재구성까지 끝낸
  **"렌더링 준비 완료" 데이터**를 한 번에 프론트로 넘긴다.
  프론트는 파싱 로직을 전혀 모르므로, 나중에 데이터 소스를 교체해도(예: 라이브러리 계측 방식 추가) 프론트는 불변이다.
- **JS 프론트**는 받은 모델을 시간축 위에서 그리는 일만 한다.
  UI 프레임워크는 **Svelte**(가볍고 Canvas 중심 앱에 군더더기 없음, Wails 공식 템플릿 지원).

### 렌더 계층 추상화
`TimelineCanvas` / `GraphCanvas`는 "모델 → 픽셀" 순수 렌더러로 설계한다.
v2에서 Canvas를 WebGL로 교체해도 인터페이스가 동일하도록 한다.

### 모듈 분리 (Go side)
- `parser` — trace 파일 → raw 이벤트
- `model` — 정규화된 goroutine / edge / 구간 타입
- `causality` — 언블록 인과 → 채널 엣지 추정
- `app` — Wails 바인딩 (얇게 유지)

## 3. 데이터 모델 & 인과 재구성

### 파이프라인
```
trace.out ──parser──▶ raw events ──model──▶ 정규화 ──causality──▶ 엣지 ──▶ 프론트 JSON
```

### Core model (Go 타입, 개념 수준)
- `Goroutine`: `id`, `createdAt`, `endedAt`, `name`(가능하면 시작 함수명), `intervals []Interval`
- `Interval`: `start`, `end`, `state`(`Running` / `Runnable(대기)` / `Blocked`),
  `blockReason`(`"chan send"`, `"chan receive"`, `"select"`, `"sync.Mutex.Lock"`, `"syscall"`, `"GC"` 등)
- `CausalEdge`: `fromG`, `toG`, `time`, `kind`(`unblock`), `inferredCategory`(`channel` / `mutex` / `other`)
- `TraceSummary`: 전체 시간 범위, goroutine 목록, 엣지 목록

### 인과 재구성 방식 (한계 포함, 정직하게)
- Go 1.22+ 새 trace 포맷의 `StateTransition`에서 **"누가 이 goroutine을 깨웠는지"**(unblocking goroutine)를
  읽어 `CausalEdge`를 생성한다.
- blockReason이 `chan send` / `chan receive`이면 그 엣지를 **`channel` 카테고리로 분류** →
  그래프 뷰에서 채널 전달처럼 그린다.
- ⚠️ **채널 고유 ID는 trace에 없다.** "G1→G2가 채널로 통신했다"까지는 보여주되,
  "어떤 채널 `ch`였는지"는 보여주지 않는다. 이 한계를 UI에서 명시한다(엣지 hover 시 "추정된 채널 통신").
- 전달된 **값**도 trace에 없으므로 표시하지 않는다.

### 프론트로 넘기는 형태
- 단일 `OpenTrace` 호출로 `TraceSummary` 전체를 JSON 직렬화해 전달한다(중간 규모라 메모리상 무리 없음).
- 프론트는 시점 `t`마다 "그 시각에 각 goroutine이 어떤 state인지 + 활성 엣지"를 모델에서 계산해 렌더한다.

## 4. 시각화 & 인터랙션 (하이브리드)

### 레이아웃
```
┌──────────────────────────────────────────────────────────┐
│  [파일 열기]   trace.out          Go 1.23   ⏮ ▶ ⏸ ⏭  1x ▾ │  ← 컨트롤바
├──────────────────────────────────────────────────────────┤
│  타임라인 (TimelineCanvas)                                 │
│  G1 ▓▓▓░░▓▓▓▓     ← 초록=실행 회색=대기 빨강=블록          │
│  G2   ▓▓▓▓░░░▓▓                                            │
│  G3      ▓▓▓▓▓                                             │
│           ┊ ← playhead(현재 t), 드래그로 스크럽            │
├──────────────────────────────────────────────────────────┤
│  라이브 그래프 (GraphCanvas) — 현재 t 시점 상태            │
│       (G1)━━send━━▶(G2⏸)        ← 노드=goroutine           │
│         ╲                          엣지=추정 채널 통신     │
│          ▶(G3)                     색=현재 state           │
└──────────────────────────────────────────────────────────┘
```

### 상단 타임라인
- goroutine당 가로 레인, X축=시간. 구간 색: 실행 / 대기 / 블록.
- 블록 구간 hover → 블록 이유 툴팁.
- playhead 수직선, 드래그로 스크럽. 휠로 시간축 줌/팬.
- 인과 엣지는 선택적으로 레인 간 곡선 화살표로 표시(많아지면 토글로 끔).

### 하단 라이브 그래프
- 현재 `t` 시점의 goroutine을 노드로, 그 부근에서 일어난 채널 인과를 엣지로 표시.
- 노드 색 = 그 시점 state(실행 / 블록).
- 레이아웃: **force-directed(d3-force)** 로 자연스럽게 배치하되, 시간 이동 시 노드가 과하게 튀지 않게 위치를 안정화.
- 재생 시 playhead가 흐르며 노드 상태·엣지가 애니메이션. 블록→언블록 순간 엣지가 "반짝"하며 전달을 표현.

### 인터랙션
- 재생 / 일시정지 / 처음·끝으로 / 배속(0.25x~4x). 스페이스 = 재생 토글.
- 타임라인 ↔ 그래프 **양방향 동기화**(타임라인 스크럽 시 그래프 갱신, 반대도).
- goroutine 클릭 시 양쪽에서 하이라이트.

### 중간 규모 대응
- goroutine이 많으면 타임라인은 **가상 스크롤**(보이는 레인만 렌더).
- 그래프는 "활성/관련 goroutine"만 표시(전부 그리지 않음).

## 5. 구현 단계화

각 단계가 독립적으로 동작·검증 가능하도록 나눈다.

- **0단계 — 스캐폴딩**: Wails 프로젝트 초기화(Svelte+TS 템플릿), `go.mod`, 빌드/실행 확인. "Hello" 창이 뜨면 완료.
- **1단계 — 파서 & 모델 (백엔드, 프론트 무관)**: `x/exp/trace`로 `trace.out` → `TraceSummary`.
  CLI/테스트로 JSON 덤프 검증. **위험의 대부분이 여기** — 먼저 단단히 한다.
- **2단계 — 인과 재구성**: 언블록 인과 → `CausalEdge`, 채널 카테고리 분류. 알려진 입력으로 단위 테스트.
- **3단계 — 타임라인 렌더 + 파일 열기**: `OpenTrace` 바인딩 → 상단 TimelineCanvas + playhead 스크럽.
  이 시점에 "trace를 열어 goroutine 구간을 본다"가 사용 가능.
- **4단계 — 라이브 그래프 + 동기화**: 하단 GraphCanvas(d3-force), 시점 상태 계산, 타임라인↔그래프 동기화.
- **5단계 — 재생 & 폴리시**: 재생/배속/애니메이션, hover 툴팁, 클릭 하이라이트, 한계 표기, 중간 규모 가상 스크롤.

## 6. 테스트 전략

- **백엔드(1·2단계)**: 표준 단위 테스트. **고정 trace 파일을 testdata로 커밋** —
  작은 Go 프로그램들(단순 send/recv, 워커풀, select, mutex, deadlock 직전)을 `go test -trace`로 미리 생성해 저장 →
  파서·인과 출력에 대한 golden 테스트. 정확성의 핵심이므로 TDD로 진행.
- **인과 재구성**: "G1 send → G2 unblock" 같은 시나리오별 단위 테스트.
- **프론트(3~5단계)**: 렌더러의 "모델 → 그릴 항목 목록" 순수 함수 부분을 단위 테스트(픽셀이 아닌 도형 데이터).
  상태 store(시점→상태) 로직도 단위 테스트. Canvas 픽셀 자체는 수동/시각 확인.
- **통합**: testdata trace를 앱으로 열어 수동 스모크 테스트.

## 7. 리스크 & 완화

- **`x/exp/trace` API 변동/문서 부족**: 1단계에서 최소 PoC를 먼저 만들어 실제 trace로 출력 확인.
- **인과 정보가 기대보다 빈약할 가능성**: 1단계 직후 실제 trace로 "인과 엣지가 충분히 나오나" 조기 검증.
  부족하면 그래프 뷰의 비중을 재조정한다.
